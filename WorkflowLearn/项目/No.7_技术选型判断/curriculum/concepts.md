# concepts

> 最近确认时间：2026-06-01
> 状态：已强化成本、build-buy-assemble、fine-tune、vendor lock-in 与 rollout 判断

## 模块 P1：先定义问题类型

- 选型之前先判断问题本质属于：生成、检索、操作、规划、判断、交互，还是混合任务。
- 如果问题本身没定义清楚，后面的技术选型一定会漂。
- AI PM 的第一责任不是先选模型，而是先把任务类型说清楚。

## 模块 P2：能力分流

- `Prompt` 适合通用生成、格式转换和轻量判断。
- `RAG` 适合有依据、可引用、会更新的知识问题。
- `Workflow` 适合步骤明确、合规要求强、路径可控的任务。
- `Agent` 适合需要动态决策、工具调用和状态推进的任务。
- `Fine-tune` 只能作为高频、稳定、可评估任务的候选项，不能默认当成“高级解法”。
- fine-tune 的成本、能力边界和替代方案变化快，进入真实方案前必须做当前核验。
- 常见分流顺序：格式不稳先看 prompt / schema；缺知识先看 RAG / grounding；步骤明确先看 workflow；需要动态判断和工具推进再看 agent；高频稳定错误且有 eval 数据时才考虑 fine-tune。

## 模块 P3：Build / Buy / Assemble

- 不是所有能力都应该从零造。
- 选型时要判断是直接购买、组合现成能力，还是自己做核心层。
- `Buy` 适合标准化、低差异、上线速度优先的能力。
- `Assemble` 适合需要把模型、工具、检索、权限和业务系统连起来的能力。
- `Build` 只适合形成长期差异化、数据护城河或关键控制权的部分。
- 真正的产品判断不是“能不能做”，而是“哪一层值得自己做，哪一层应该借力”。
- 供应商锁定要作为正式选型维度：API 语义、prompt 架构、向量库数据格式、eval 管线、日志结构、权限模型和团队运营习惯都会产生切换成本。
- 可接受锁定的前提是收益明确、替代路径清楚、数据可导出、关键评估样本可迁移；否则锁定应进入风险清单或一票否决。

## 模块 P4：成本、风险与维护

- 技术方案的成本不止模型调用费，还包括工具调用、搜索/检索、缓存、存储、人工复核、评估、监控和组织协调。
- token 成本只是 unit economics 的一层；高频工具调用、外部搜索、向量检索、上下文缓存和日志存储都可能改变成本结构。
- 成本模型要同时看单次任务成本、月度规模成本、峰值成本和人工兜底成本。
- 风险不止是效果差，还包括可解释性差、升级代价高、接管困难和长期漂移。
- 维护成本决定方案能不能长期跑，不是 demo 好不好看。
- 切换成本也是维护成本：如果换模型、换向量库或换 agent 平台会重写 prompt、schema、eval 和运营流程，就必须提前写进方案评审。

## 模块 P5：ROI 与商业化

- ROI 不是“感觉有用”，而是要能拆成节省时间、提升转化、减少错误、降低人工和扩大覆盖。
- B 端产品常见的定价方式包括 seat、usage、outcome、free-tier 和混合定价。
- AI 原生定价要看边际成本、用户价值单位和风险责任：有的能力适合按席位，有的适合按任务量，有的适合按结果或节省成本分成。
- 选型判断最终要回到商业可行性：是否值得做，是否卖得出去，是否撑得住长期成本。

## 模块 P6：产品化决策

- 产品化不只是把模型接上去，而是把失败、接管、解释和反馈机制一起设计进去。
- 选型时必须同时考虑 UI、SOP、权限、日志、人工复核和上线门槛。
- rollout 决策要说明从 demo 到 pilot、灰度、扩大上线的触发条件、停止条件和回滚方案。
- 好方案的标准不是最先进，而是最适合当前约束和目标。

## 模块 P7：演进路径

- 很多系统不是一步到位，而是从 prompt 开始，逐步演进到 RAG、workflow、agent 和更复杂的编排。
- 课程里要学的是“演进判断”，而不是死记某个阶段的静态答案。

## 模块 P8：AI PM 跨职能协作

- AI PM 要能和 ML 工程、后端、前端、数据、法务、安全、运营和 GTM 对齐同一套目标、风险和上线门槛。
- 与 ML 工程协作看 eval 和数据；与工程协作看接口、状态、权限和可观测性；与法务/安全协作看风险等级和合规义务；与运营协作看接管和 SOP。
- 好的选型 memo 应明确 owner、决策输入、验收证据和争议处理，而不是只列技术优缺点。

## 当前不进入稳定层的内容

- 具体 vendor shortlist 与折扣条款
- 具体组织分工模板
