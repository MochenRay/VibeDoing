# 掌握卡片/02. 提交三部曲.md

## 我的理解
Git 的存档过程不是“一键保存”，而是分三步走的“精细化操作”：
1.  **Add (暂存)**：选商品进购物车（或者放进篮子）。这是一个待打包的状态。
2.  **Commit (提交)**：把购物车里的东西下单封箱，生成一个正式的订单（存档）。
3.  **Push (推送)**：把包裹发给云端（GitHub）。

**核心价值**：`add` 的存在是为了让我们能分批处理变更（原子提交）。比如修 Bug 和改错别字可以分成两个包裹发货，这样以后退货（回滚）时互不影响。

## 🏀 类比挂钩
*导师与我的对话精华*
我：用网购来举例，add 就好像把文件的变动存进一个篮子，我可以多次 add 多件商品，然后 commit 来把它打包贴上标签... 分两次打包是为了方便我以后更好地定位或者退货... 如果两个篮子的东西放在一起，那点退货就会把有用的也退回去了。
导师：`git commit --amend` 就像是拆开刚才的包，把漏掉的东西塞进去，换张新快递单重新封好。

## 踩过的坑
- **误区**：以为 `git add .` 是万能的。
- **真相**：这会把垃圾文件（如 `test_junk.log`）或者不想提交的修改一并打包。必须用 `.gitignore` 给垃圾文件穿上隐身衣，或者手动 add 指定文件。
- **误区**：以为 `commit` 就是把变动发给云端了。
- **真相**：`commit` 只是本地封箱。必须 `push origin main` 才会发给云端。

## 源真理定义
> **Atomic Commit (原子提交)**：一种最佳实践，即每个提交（Commit）应该只包含一个独立的逻辑变更。
> **Staging Area (暂存区)**：Git 目录下的一个文件（索引），保存了下次将要提交的文件列表信息。
