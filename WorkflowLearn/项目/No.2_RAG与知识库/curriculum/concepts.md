## 模块 P1：多源数据 vs 多模态

- 多源数据强调“来自不同系统或来源”，多模态强调“来自不同感知通道或表示形式”。
- 两者都在解决同一个底层问题：如何让不同类型的数据协同工作并产生更完整的判断。
- 这层桥接很重要，因为它能把原有的数据融合经验翻译成 AI 语言，而不是从零开始学 RAG。

## 模块 P2：ETL vs AI Pipeline

- 传统 `ETL` 服务的是结构化存储和分析，AI Pipeline 在其基础上增加了标注、向量化、索引等语义层步骤。
- `RAG` 的“分块 -> 向量化 -> 检索”就是典型 AI Pipeline，而不是单纯数据库查询。
- 产品上最重要的转变，是从“字段精确匹配”转到“语义表示和语义召回”。

## 模块 P3：数据质量对 RAG 的影响

- 数据质量问题在 RAG 里常常是静默失效，系统看起来工作正常，但回答并不可靠。
- 传统数据治理里的完整性、准确性、一致性、时效性，仍然能直接映射到 RAG。
- 典型表现包括文档缺失、内容过时、表述不一致和分块断裂。
- 这意味着 RAG 失败时，不能只盯模型，还要看文档治理、版本、切分与知识覆盖。

## 模块 A1：RAG 流程与架构

- `RAG` 的直觉是“先检索，再生成”，核心价值在于把企业或私有知识接入模型回答。
- 离线阶段通常是文档处理、分块、向量化和建索引；在线阶段通常是问题改写、检索、拼接上下文和生成回答。
- 与纯大模型相比，RAG 的优势主要体现在私有知识、知识更新、可追溯性和长文档利用效率。

## 模块 A2：向量与 Embedding

- `Embedding` 模型负责把文本变成向量，`LLM` 负责理解和生成；两者不是一个东西。
- 向量是语义表示的结果，Embedding 是得到这个表示的过程。
- 语义相近的文本在向量空间里距离更近，因此检索可以突破纯关键词匹配的限制。
- 对产品经理最重要的意义不是推公式，而是理解“召回失败可能来自语义覆盖不足，而不只是没写关键词”。

## 模块 B1：文档分块策略

- 分块决定知识以什么粒度进入索引，是 RAG 效果的关键前置条件。
- 核心矛盾是精度与完整性的权衡：切太大噪声多，切太小上下文断。
- 分块策略不只是“怎么切”，还包括切完后是否补充元数据、摘要和适用场景描述。
- 结构化文档、SOP、FAQ、案例库应采用不同切分单位，而不是一套固定参数走天下。

## 模块 B2：检索错误 vs 生成错误

- RAG 系统出错时，先定位问题发生在哪个环节，再谈优化，否则优化方向会错。
- 检索错误是“找错了或没找到”，生成错误是“找对了但答错了”。
- 诊断顺序应该先看检索到的文档是否匹配用户意图，再看回答是否准确反映了这些文档。
- 对产品设计来说，引用高亮、检索结果透出、反馈按钮和后台诊断面板都属于高价值能力。

## 模块 B3：RAG 适用边界

- B3 的目标不是证明“RAG 很强”，而是训练能复用的 `RAG fit memo`：先判断问题类型，再决定是否建检索系统。
- RAG 适合：答案必须来自既有文档或知识库、知识会更新、用户需要出处、文档体量或表达差异让人工搜索成本较高。
- RAG 不适合：实时计算、强事务查询、复杂多步规划、固定流程审批、纯创意生成，或数据量小到索引维护成本高于收益的场景。
- 长上下文更适合一次性读少量完整材料；搜索更适合公开网页或实时信息；工具调用更适合计算、查询结构化系统或执行动作；workflow 更适合稳定的多步骤业务流。
- 诊断失败时要拆层：没召回相关依据是 retrieval failure；依据正确但回答误读是 generation failure；需要查库/计算/写入却只做文本回答是 tool failure；步骤顺序、接管和状态流错位是 workflow failure。
- 一条实用判断线是：问题本质是在“找已有知识并给出可追溯回答”，还是在“读完整材料 / 搜实时信息 / 做运算 / 调工具 / 跑流程”。
