## 模块 P1：Prompt 是接口，不是技巧

### 第 1 轮

- 讲 prompt 的本质不是“哄模型”，而是定义输入、约束、输出和失败边界。
- 正例：一个好 prompt 让下游系统知道结果长什么样，而不是只得到一段好看的话。

### 第 2 轮

- 讲为什么产品语境里 prompt 更像接口合同。
- 讲如果 prompt 不能测、不能复用、不能审计，就不是好接口。

### 理解确认问题

- 为什么把 prompt 当“文案技巧”会把产品带偏？

## 模块 P2：System / User 层级与边界

### 第 1 轮

- 讲 system 层放长期规则，user 层放当前任务。
- 正例：安全边界和输出规则应进 system，不该每次重复塞进 user prompt。

### 第 2 轮

- 讲层级分离如何减少注入风险和上下文污染。
- 讲为什么所有长期规则都堆在 user prompt 里会很脆。

### 理解确认问题

- 什么类型的规则一定要放 system 层？

## 模块 P3：Prompt 基本结构

### 第 1 轮

- 讲角色、任务、输入、输出格式、约束、示例是常见骨架。
- 讲目的不是写模板，而是让模型更稳定执行。

### 第 2 轮

- 讲结构化 prompt 如何减少歧义。
- 讲“写很长”不等于“写清楚”。

### 理解确认问题

- 为什么清晰约束常常比冗长说明更重要？

## 模块 P4：Few-shot 与示例设计

### 第 1 轮

- 讲 few-shot 是给模式，不是灌知识。
- 正例：给一好一坏两个示例，比给十个同质示例更有用。

### 第 2 轮

- 讲示例应该覆盖正常、边界和易错情况。
- 讲重复示例如何放大偏差。

### 理解确认问题

- 为什么“示例越多越好”通常是错的？

## 模块 P5：Schema-first 与结构化输出

### 第 1 轮

- 讲只要下游系统要处理输出，就要优先 schema-first。
- 正例：抽取结构化字段、生成 UI 描述、调用工具参数。

### 第 2 轮

- 讲 schema 的价值是减少歧义、降低解析成本、便于验证和回放。
- 讲 schema 不只等于 JSON，也可以是表格、枚举和界面结构。

### 理解确认问题

- 为什么“先出自由文本再解析”常常不是好产品方案？

## 模块 P6：Tools 与 Tool Contracts

### 第 1 轮

- 讲 tool use 是把模型从只会说，变成会调用外部动作。
- 正例：查数据库、发起审批、搜索资料，这些都需要明确 contract。

### 第 2 轮

- 讲 contract 要定义参数、返回值、错误、幂等性和副作用边界。
- 讲工具不是越多越好，边界清晰更重要。

### 理解确认问题

- 为什么一个模糊的工具接口会直接放大 agent 不稳定性？

## 模块 P7：Prompt 作为工作流编排语言

### 第 1 轮

- 讲跨步骤任务里，prompt 不再只是单轮回答说明书，而是流程推进语言。
- 正例：让系统先抽取、再判断、再调用工具、再写回状态。

### 第 2 轮

- 讲状态传递、输出校验、失败重试和交接如何一起构成可靠 workflow。
- 讲为什么长 prompt 不等于编排。

### 理解确认问题

- 什么情况下你需要 workflow 设计，而不是继续在一个 prompt 上打补丁？

## 模块 P8：Generative UI / AI-native 交互

### 第 1 轮

- 讲 AI 输出不只可以是文本，也可以是表单、卡片、步骤流和局部界面。
- 正例：把结果直接渲染成可编辑草稿，比回一段说明更适合工作流。

### 第 2 轮

- 讲 generative UI 的重点是可渲染、可操作、可回退。
- 讲为什么“聊天框万能论”会限制 AI 产品设计。

### 理解确认问题

- 什么样的任务更适合 generative UI，而不是 chat？

## 模块 P9：Prompt Quality Principles

### 第 1 轮

- 讲好 prompt 的标准是稳定、可测、可复用、可审计。
- 讲要学会区分 prompt 问题、schema 问题、tool 问题和模型边界问题。

### 第 2 轮

- 讲失败归因：回答差，到底是需求没写清、格式没定义、工具没控好，还是模型本身不够。

### 理解确认问题

- 你怎么判断一次失败该改 prompt，还是该改 schema / tool / model？
