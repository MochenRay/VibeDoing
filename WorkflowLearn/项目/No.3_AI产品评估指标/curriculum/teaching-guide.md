## 模块 P1：为什么 AI PM 需要 evals

### 第 1 轮

- 讲 eval 不是“上线后看看数据”，而是把需求、验收和放量绑成一个合同。
- 正例：两个 demo 都很好看，但只有定义了目标、样本和指标，团队才知道哪个更值得上线。

### 第 2 轮

- 讲没有 eval 时，团队通常在比较感觉，而不是比较系统质量。
- 讲 PM 的职责是把“看起来不错”翻译成可验证标准。

### 理解确认问题

- 为什么 AI PM 不能只靠 demo 观感做判断？

## 模块 A1：混淆矩阵与基础分类指标

### 第 1 轮

- 讲 `TP/TN/FP/FN` 是理解误报和漏报的基本语言。
- 正例：风控审核里，误报和漏报的代价完全不同。

### 第 2 轮

- 讲 precision、recall、F1 分别适合什么，不适合什么。
- 强调：指标不是业务代价本身，只是描述切片。

### 理解确认问题

- 为什么 F1 高不等于业务就一定好？

## 模块 A2：评估设计四步

### 第 1 轮

- 讲主线：`objective -> dataset -> metrics -> compare / iterate -> continuous eval`。
- 直觉抓手：先定成功，再定样本，再定衡量。

### 第 2 轮

- 讲 dataset 不能只收漂亮样本，必须纳入边界样本、失败样本和真实生产样本。
- 讲 metric 不能只靠一个分数，通常要组合自动指标、人工判断和业务指标。

### 理解确认问题

- 为什么“先收一堆数据再说”通常会把 eval 做歪？

## 模块 B1：Workflow / Retrieval Evals

### 第 1 轮

- 讲多步系统不能只看最终答案，因为前面任一步错了，后面都可能被污染。
- 正例：RAG 回答错时，先检查检索到了什么。

### 第 2 轮

- 讲 retrieval eval 要看召回、引用对应、上下文充分性。
- 讲 workflow eval 要拆关键步骤，而不是只看端到端好不好。

### 理解确认问题

- 为什么一个“最终答案分”不足以指导 workflow 系统优化？

## 模块 B2：Agent Evals

### 第 1 轮

- 讲 agent 的问题不只在内容正确，还在工具选择、参数、时机和交接。
- 正例：任务最终完成了，但路径极不稳定，这在 agent 里仍是风险。

### 第 2 轮

- 讲要评工具选用、状态推进、增量进展和失败恢复。
- 讲长流程 agent 不能只看完成率。

### 理解确认问题

- 为什么 agent eval 比单轮问答 eval 更像系统评估？

## 模块 B3：Launch Readiness

### 第 1 轮

- 讲上线前至少要回答：离线够不够、护栏稳不稳、失败怎么接管、上线后怎么持续看。
- 正例：离线过线但没有回滚和人工接管，不算可上线。

### 第 2 轮

- 讲小流量验证和护栏指标的重要性。
- 讲 A/B test 不能替代离线质量验证。

### 理解确认问题

- 为什么 AI 产品上线判断比普通功能更依赖分层 gate？
