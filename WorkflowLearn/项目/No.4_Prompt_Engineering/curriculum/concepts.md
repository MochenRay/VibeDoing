# concepts

## 说明

本文件承接跨时间仍成立的 Prompt / Schema / Tools 基础概念。
它不是 prompt 话术合集，而是 AI 产品接口设计的基础课。

## 模块 P1：Prompt 是接口，不是技巧

- Prompt 的本质是定义输入、约束、输出和失败边界。
- 对 AI 产品来说，Prompt 不是“哄模型”，而是“把人类意图翻译成机器合同”。
- Prompt 设计得好不好，直接决定产品是否可测、可控、可交付。

## 模块 P2：System / User 层级与边界

- System Prompt 负责定义身份、边界、行为规则和优先级。
- User Prompt 负责表达当前任务，不应承载全部长期约束。
- 层级分离的价值在于减少注入风险、减少上下文污染、提高复用性。
- 任何需要持续生效的产品规则，都应该优先进入系统层或宿主层配置，而不是散落在用户提示里。

## 模块 P3：Prompt 基本结构

- 一个完整的 Prompt 通常包含：角色、任务、输入、输出格式、约束、示例。
- 结构化写法的重点不是“写得像模板”，而是让模型更容易稳定执行。
- 清晰度、显式约束和可验证输出，比长篇说明更重要。

## 模块 P4：Few-shot 与示例设计

- few-shot 的作用不是灌输知识，而是给模型一个可模仿的决策模式。
- 示例要覆盖正常情况、边界情况和容易出错的情况。
- 示例质量比示例数量更重要；重复、单一、同质的示例只会放大偏差。

## 模块 P5：Schema-first 与结构化输出

- 只要下游系统需要处理输出，就应该优先把输出定义成 schema。
- 结构化输出的价值是减少解析成本、降低歧义、便于验证和回放。
- schema-first 不是只适合 JSON；它也可以是表格、枚举、树结构、UI 指令或工具参数。

## 模块 P6：Tools 与 Tool Contracts

- 工具调用的本质是把模型从“只会说”扩展为“可执行动作的编排者”。
- Tool Contract 要定义清楚：工具名、参数、返回值、失败条件、幂等性和副作用边界。
- 不是工具越多越好，而是工具边界越清楚越好。
- 工具是否可用，不能只看模型会不会发起调用，还要看参数是否正确、失败是否可恢复、日志是否能支撑复盘。

## 模块 P7：Prompt 作为工作流编排语言

- 当任务跨步骤、跨状态、跨工具时，Prompt 需要服务于 workflow，而不只是单轮回答。
- 可靠的编排通常需要：任务分解、状态传递、输出校验、失败重试和交接。
- 对 Agent / Workflow PM 来说，Prompt 的目标不是生成长文，而是驱动可恢复的流程。

## 模块 P8：Generative UI / AI-native 交互

- AI 产品的输出不一定是文本；它也可以是表单、卡片、步骤流、草稿、图表或局部可编辑界面。
- Generative UI 的核心是让模型输出可渲染、可操作、可回退的界面结构。
- AI-native 交互的重点是把不确定性前置到界面层，而不是把错误全部塞进答案文本里。
- 只要 AI 输出会驱动界面或动作，就要同步定义 schema 校验、用户确认、失败状态和可回放证据。

## 模块 P9：Prompt / Schema / Tools 的评估闭环

- Prompt 质量不能只靠主观感觉，要通过代表性样本、失败样本和回归样本持续验证。
- Schema 要评估字段完整性、枚举正确性、边界输入和下游解析稳定性。
- Tool use 要评估工具选择、参数正确性、错误处理和副作用控制。
- 对 AI PM 来说，prompt 不是一次性文案，而是可测试、可追踪、可迭代的行为规格。

## 模块 P10：Prompt Quality Principles

- 好 prompt 的标准不是“漂亮”，而是稳定、可测、可复用、可审计。
- 模糊要求会逼模型自由发挥；清晰约束会逼模型执行。
- 产品经理需要学会区分：是 prompt 问题、schema 问题、tool 问题，还是模型能力边界问题。
