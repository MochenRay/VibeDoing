# 掌握卡片：C1. Issues

## 🧠 我的理解
- **现状痛点**：目前使用 OmniFocus/Plan (规划) + Coding (执行)。需手动在两个系统间同步状态，自己充当了"人肉接口" (Human API)。
- **GitHub价值**：打通代码与任务。

## 🔗 类比挂钩
**"便利贴墙"**
- Open 是墙上的纸，Closed 是揉掉的纸。
- 只有贴在墙上的（GitHub Issues）大家才看得到，仅仅写在自己笔记本里的（OmniFocus）不算数。

## 🚫 踩过的坑
- **错误**：以为 GitHub 全知全能，会自动把修好的代码和 Issue 对应起来。
- **真相**：它是个笨机器人。必须在 PR 里写明 `Fixes #42` (咒语)，否则代码合并了 Issue 依然是 Open 的。
- **教训**：自动化需要"显式指令"。

## 📖 源真理定义
> **Issue (议题)**：GitHub 上的任务追踪单元。它不仅仅是 Bug 单，而是“任何需要讨论或追踪的事情”的容器（Feature、Bug、Question、Idea）。

## 💡 验证高光
- **自动化**：必须显式使用 `Fixes #ID` 才能触发状态流转。
- **溯源性 (Traceability)**：Issue 是一个 "Context 容器"。因为它链接了 PR，所以你永远能从 Issue 里点进代码去找"当年是谁、改了哪行、为了修这个 Bug"。不再需要去翻几十页的聊天记录。
