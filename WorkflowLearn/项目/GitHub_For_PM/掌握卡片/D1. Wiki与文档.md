# 掌握卡片：D1. Wiki与文档

## 🧠 我的理解
- **核心理念**：Docs as Code。文档不是游离的附件，而是代码仓库的一等公民。
- **分支即版本**：想看线上的文档就切 `main`，想看开发中的就切 `feature`。永远不会搞错"哪个版本对应哪个功能"。

## 🔗 类比挂钩
**"Snapshot vs Stream"**
- Word/飞书是 **Snapshot (快照)**，发出来就死在那里了，容易过期。
- GitHub Docs 是 **Stream (流)**，它随着代码分支的流动而实时变形。

## 🚫 踩过的坑
- **误区**：强迫非技术干系人（如老板）去 GitHub 看 Markdown 文档。
- **真相**：**文档分层 (Layered Documentation)**。GitHub Docs 是给"干活的人"（PM/Dev/QA）看的，追求准确和版本控制；给老板应该单独做 PPT 或简报，追求可读性和结果。

## 📖 源真理定义
> **Docs as Code (文档即代码)**：将文档视为代码的一部分，使用相同的工具（Git）、相同的流程（PR）和相同的位置（Repo）进行管理。

## 💡 验证高光
- **文档分层**：对上（老板）用 PPT/Word 讲故事；对内（团队）用 Markdown 讲逻辑。
- **技术壁垒**：为什么必须用 Markdown？因为 Word 是 **二进制文件 (Binary)**，Git 把整个 `.docx` 视为一坨乱码，无法进行行级 Diff，也无法合并冲突。Markdown 是 **纯文本 (Plain Text)**，Git 能精确管理每一个字符的变更。这是 "Docs as Code" 的物理基础。
