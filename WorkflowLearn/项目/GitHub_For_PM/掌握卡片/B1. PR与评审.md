# 掌握卡片：B1. PR与评审

## 🧠 我的理解
- **PR的本质**：不是单纯的代码合并，而是关于代码的对话会议。
- **核心价值**：它是白盒审计，防止开发用硬编码（Hardcode）糊弄验收。Staging 只能验证现在的表现，PR 验证未来的健壮性。

## 🔗 类比挂钩
**"出版社审稿"**
- 作者不仅提交章节，还要写申请。
- 编辑（PM）在旁边写批注，批准后才能印刷。

**"建筑验收"**
- Tech Lead 敲墙（查代码质量）。
- PM 拿设计图看窗户位置（查业务逻辑）。

## 🚫 踩过的坑
- **错误**：认为截图和录屏就能验收。
- **原因**：过于信任黑盒表现（Staging），忽视了内部逻辑。
- **教训**：必须看 PR，即使不懂代码也能看懂是否 Hardcode。

## 📖 源真理定义
> PR 是开发请求将“野史”（Feature Branch）并入“正史”（Main Branch）的过程。

## 💡 验证高光
- **场景判断**：开发本地改代码 push 上来，PR 是否会自动更新？
- **你的推理**：PR 对应的是一个分支，分支的后续 commit 也会更新到 PR 上。如果不自动更新，只是因为浏览器没刷新。
- **正解**：**你会看到动态更新**。GitHub 的 PR 页面是实时的（Real-time），只要分支有新提交，页面上会立刻弹出一个黄色条提示 "Updates were made to the base branch"，或者直接刷新 Files Changed。
- **本质**：**PR 不是一份静态文件，而是一个监控分支动态的窗口**。
