# 掌握卡片：B2. 冲突处理

## 🧠 我的理解
- **冲突本质**：不是 Bug，而是两个正确的逻辑在时空中撞车了。需要人来做“裁决者”。

## 🔗 类比挂钩
**"腾讯文档协同编辑"**
- A改了标题，B也改了标题，同时保存时，系统不知道听谁的，只能弹窗让人生效。

## 💡 个人顿悟
- **隐性冲突**：Git 只管物理行冲突（全绿），不管业务逻辑冲突（炸雷）。
- **解决直觉**：需要"上下文追溯"。如果改了一行代码，应该能快捷查看"谁引用了我"，逐级展开影响范围。（这精准对应了 IDE 的 Find Usages 功能）。

## 🚫 行业常见陷阱 (已避开)
- **警示**：虽然你在验证环节成功**识破**了开发的"强推"建议（选了 B），但这依然是 PM 最容易翻车的点。
- **错误行为**：允许开发为省事直接"覆盖"（Overwrite）冲突代码。
- **真相**：冲突是时空碰撞，必须"融合"（Merge）。"覆盖"等于删除了同事在平行宇宙里的工作成果，是重罪。
- **直觉防线**：听到"强推"、"强制覆盖"要警铃大作。

## 📖 源真理定义
> **Merge Conflict (合并冲突)**：当两个分支对**同一文件的同一行**进行了不同的修改时，Git 无法自动决定保留哪一个，必须由人来裁决。
