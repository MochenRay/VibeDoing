## 模块 P1：Agent 是什么

### 第 1 轮

- 讲 agent 不是会聊天，而是围绕目标推进任务的系统。
- 直觉抓手：聊天模型在“回答”，agent 在“完成”。

### 第 2 轮

- 讲目标导向、状态推进、工具调用和任务完成是关键判断线。
- 讲为什么很多产品把“有工具调用的聊天框”误叫 agent。

### 理解确认问题

- 一个系统满足什么条件后，你才愿意叫它 agent？

## 模块 P2：Workflow 与 Agentic Planning

### 第 1 轮

- 讲 workflow 重显式编排，agentic planning 重动态决策。
- 正例：固定审批链更像 workflow；开放调研与计划更可能需要 agentic planning。

### 第 2 轮

- 讲不是二选一，而是哪里该强约束，哪里该放权。
- 讲产品判断核心是风险、可控性和收益，而不是概念新旧。

### 理解确认问题

- 什么时候你会坚决不用自由 agent，而优先 workflow？

## 模块 P3：ReAct 与思考-行动-观察循环

### 第 1 轮

- 讲 reasoning、acting、observing 组成 agent 最小闭环。
- 正例：先判断该查什么，再调用工具，再根据结果修正下一步。

### 第 2 轮

- 讲为什么没有观察和回写，就谈不上真正闭环。
- 讲这个循环如何影响延迟、错误率和接管设计。

### 理解确认问题

- 为什么“会调用一次工具”不等于已经具备 agent loop？

## 模块 P4：Tool Use 与 Contract

### 第 1 轮

- 讲工具是外部动作边界，不是插件名词。
- 正例：搜索、写数据库、发消息、审批请求，都需要清楚的 contract。

### 第 2 轮

- 讲参数清晰、错误可处理、权限可控的重要性。
- 讲工具粒度过粗和过细都会伤害可调试性。

### 理解确认问题

- 为什么 tool contract 是 agent 稳定性的核心，而不是附属文档？

## 模块 P5：State / Memory / Artifact

### 第 1 轮

- 讲长流程任务如果没有状态，就只能每轮重来。
- 直觉抓手：state 是当前进展，memory 是可复用信息，artifact 是交接凭证。

### 第 2 轮

- 讲三者区别和联系。
- 讲为什么 artifact 对人工接管和恢复特别重要。

### 理解确认问题

- 为什么“把所有内容都堆进上下文”不能替代 state 设计？

## 模块 P6：Harness 与长流程执行

### 第 1 轮

- 讲 harness 是让长流程稳定跑的外层控制，不是模型自带能力。
- 正例：初始化、检查点、回退、超时、收口，都属于 harness。

### 第 2 轮

- 讲为什么 demo 能跑不代表 production 能跑。
- 讲 harness 是把 agent 从一段炫技演示变成可交付系统的关键。

### 理解确认问题

- 为什么长流程 agent 不能只靠 prompt 自己“悟”出可恢复性？

## 模块 P7：Observability 与 Reliability

### 第 1 轮

- 讲不只看最终答案，还要看工具调用、状态变化、失败位置和恢复轨迹。
- 正例：最终失败只是一层表象，真正要定位的是哪一步开始偏了。

### 第 2 轮

- 讲 observability 的目标是可定位、可归因、可复盘。
- 讲 reliability 不是永不出错，而是出错后仍可控。

### 理解确认问题

- 为什么“最后结果对了”仍然可能隐藏可靠性问题？

## 模块 P8：多 Agent 与组织落地

### 第 1 轮

- 讲多 agent 的本质是角色分工和交接协议，不是“越多越高级”。
- 正例：明确拆成规划者、执行者、审核者，才可能有收益。

### 第 2 轮

- 讲组织流程、人工复核、审批点和 SOP 改造常常比多 agent 架构更重要。
- 讲为什么很多多 agent 系统只是把复杂度翻倍。

### 理解确认问题

- 什么情况下你会明确反对上多 agent？
