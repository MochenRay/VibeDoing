# RAG Fit Memo

> 最近确认时间：2026-06-01
> 场景：城市治理知识助手
> 结论：建议采用 `RAG + grounding + citations + fallback`

## 问题定义

目标用户需要快速查询：

- 政策法规
- 办事指南
- 操作手册
- 历史处置案例

当前痛点不是“没有答案”，而是：

- 文档分散在多个系统
- 关键词不统一
- 政策会更新
- 用户必须知道出处才能继续执行

## 为什么适合 RAG

- 问题本质是在找已有知识，而不是做复杂运算
- 知识会持续更新，不适合只靠模型参数记忆
- 回答必须可追溯，不能只给口头结论
- 文档体量较大，且表达常和用户提问不一致

## 替代方案边界

- 长上下文：适合一次性审阅少量完整材料；不适合作为持续更新的多源知识入口
- 搜索：适合公开网页或实时信息；不适合直接承接内部文档治理、版本控制和引用验收
- tool use：适合查结构化系统、计算或执行动作；本场景的主问题仍是解释既有政策与案例
- workflow：适合固定审批和表单流；本场景的问题表达开放，不能只靠预设流程覆盖

## 为什么不选纯模型

- 纯模型无法保证基于最新政策
- 无法天然给出可核对的出处
- 高风险问题里，答得像真的比答不出来更危险

## 为什么不选纯工具 / 纯 workflow

- 纯关键词检索覆盖不足，容易漏掉“表达转换”问题
- 纯规则 workflow 很难覆盖自然语言提问变化
- 仍然需要模型把多段依据组织成用户可读回答

## 什么时候不该上 RAG

- 任务本质是实时计算
- 任务依赖事务型系统写入与审批
- 数据量很小，维护索引成本高于收益
- 用户需要的是固定表单流，而不是知识问答

## 风险与前置条件

- 如果知识源质量差，RAG 会静默失效
- 如果分块过碎或过粗，引用体验会变差
- 如果没有 citation 和 fallback，用户会把它误当成“会编造的搜索框”

## 失败归因边界

- `retrieval failure`：没有召回对口政策、SOP 或案例
- `generation failure`：依据召回正确，但回答误读、漏读或过度概括
- `tool failure`：问题需要实时查库、计算或提交动作，却被误当成知识问答
- `workflow failure`：需要审批、接管、状态流转或多步编排，但产品只返回文本答案

## 决策

- 主方案：`RAG + grounding + citations`
- 配套要求：
  - 必须展示出处
  - 必须暴露版本/发布日期
  - 必须有“无法确认时不直接回答”的 fallback
  - 必须保留人工接手路径
