# AI PM 全景预期产出清单

> 最近确认时间：2026-04-16
> 状态：正式输出清单

## 目的

这份清单定义你在 `AI 应用 PM + Agent / Workflow PM` 学习过程中，最终应该产出的完整作品版图。
它不等于“学完多少课”，而是定义“你学完后手里应该拿得出什么”。

## 使用原则

- 每个模块至少有一个可验证产出，不做纯阅读式学习。
- 产出分为四层：`理解证明`、`设计文档`、`评估方案`、`求职作品`。
- 价格快照、模型榜单、短期 benchmark 不是最终产出。
- 最终作品必须能够支持求职面试、作品集展示和真实场景讨论。

## A. 共同主干产出

### M1 LLM 基础边界

- `LLM 边界说明稿`
  - 能向非技术面试官解释 token、context、sampling、hallucination、LLM 的工作边界
- `能力边界判断清单`
  - 能判断一个需求更适合纯模型、RAG、tool use 还是 agent

### M2 Prompt / Schema / Tool Use

- `Prompt 设计 brief`
  - 包含 system prompt、user prompt、few-shot、约束和失败处理
- `结构化输出 schema`
  - 至少一份稳定的 JSON schema / tool contract
- `工具调用设计稿`
  - 说明什么时候该让模型说话，什么时候该调工具

### M3 Retrieval / Grounding

- `RAG 架构简图`
  - 离线处理、在线检索、生成、诊断路径
- `知识库切分策略稿`
  - 按文档类型给出 chunking / metadata / source design
- `RAG 失败归因单`
  - 能区分 retrieval failure 与 generation failure

### M4 Evals / Launch Readiness

- `AI feature eval plan`
  - 包含 objective、dataset、metrics、judge / human review、go / no-go
- `Workflow / RAG scorecard`
  - 检索层、生成层、护栏指标分层设计
- `Agent launch gate`
  - 离线指标、在线护栏、人工接管和回滚条件

### M5 Fallback / HITL / Trust

- `Fallback 分级方案`
  - AI 回答、规则/知识库兜底、转人工
- `HITL 边界设计`
  - 哪些节点 AI 自动过，哪些节点必须人工确认
- `信任与解释策略稿`
  - 引用展示、置信度表达、失败提示、追问和纠错入口

## B. 系统化加深产出

### M6 Agent Fundamentals

- `Agent 任务分解稿`
  - 说明单 agent、workflow、multi-agent 的取舍
- `Agent 能力边界图`
  - 什么适合 agent，什么不适合

### M7 State / Memory / Long-running Tasks

- `状态设计稿`
  - 任务状态、记忆层、handoff artifact、恢复策略
- `长流程任务恢复方案`
  - session 断裂后如何继续、如何防止“假完成”

### M8 MCP / Tool Contracts / Connectors

- `Tool contract 套件`
  - 一组有 schema、有错误处理、有返回规范的工具定义
- `Connector / MCP 适配判断稿`
  - 何时接系统，何时接 API，何时保持人工步骤

### M9 Runtime / Observability / Reliability

- `Agent runtime 观测面板草图`
  - status、latency、tool errors、handoff、retry、fallback
- `故障分类手册`
  - prompt、tool、retrieval、state、policy、human handoff 等错误类型
- `反馈数据闭环设计`
  - 显式反馈、隐式反馈、失败日志如何回流到 eval dataset

## C. 产品化与作品产出

### M10 AI 功能设计与体验

- `Generative UI 方案稿`
  - copilot、inline edit、ghost text、AI panel 或 tool-driven UI
- `失败体验设计稿`
  - 模型答错、工具失败、检索为空时的 UX 降级方案
- `AI-native 交互规范`
  - 哪些地方用 chat，哪些地方不用 chat

### M11 垂直场景方案

- `垂直场景 PRD`
  - 例如客服、知识助手、运营分析、销售辅助、内部 workflow
- `ROI 估算稿`
  - 单次任务成本、人工替代/辅助价值、可衡量业务收益
- `组织落地方案`
  - SOP 改造、人工审批位、责任边界、上线节奏

### M12 AI 应用 PM 作品

- `作品 1：AI feature PRD`
- `作品 2：RAG / eval / fallback 一体化设计`
- `作品 3：Generative UI 体验方案`
- 每个作品都要包含：
  - 问题定义
  - 用户流程
  - 系统结构
  - 评估方案
  - 风险与 fallback
  - 上线策略

### M13 Agent / Workflow PM 作品

- `作品 1：Workflow agent PRD`
- `作品 2：Tool / MCP / state 设计稿`
- `作品 3：Agent launch gate + observability + handoff 方案`
- 每个作品都要包含：
  - 任务编排逻辑
  - 工具边界
  - 人机协作位
  - 失败恢复
  - 评估与运行指标

## D. 最终求职包

### 面试回答资产

- `AI PM 核心问题标准答法`
  - 为什么适合做 AI PM
  - 如何判断一个 AI 功能是否该做
  - 如何设计 fallback / HITL
  - 如何做 eval 和 launch gate
  - 如何判断 agent 是否真的值得上

### 作品集资产

- `1 页总览`
  - 你的方向、方法论、代表作品
- `3-5 个重点案例`
  - 至少 2 个 AI 应用 PM 案例
  - 至少 1-2 个 workflow / agent 案例
- `附录资产`
  - eval brief
  - tool contract
  - workflow diagram
  - launch checklist

### 展示能力

- 能向业务方解释价值
- 能向技术方解释架构边界
- 能向面试官展示你不是“会写提示词”，而是“会设计可上线的 AI 系统”

## E. 完成标准

以下条件同时满足，才算这条路线真正完成：

- `No.1-No.8` 都有新 schema 下的正式课程落点
- 每个大模块至少有一个代表性产出
- 至少形成 `3` 个完整作品
- 至少形成 `1` 套可复用的 eval / launch 模板
- 至少形成 `1` 套可复用的 tool / workflow / handoff 设计模板
- 作品能够覆盖 `AI 应用 PM` 和 `Agent / Workflow PM` 两个方向
