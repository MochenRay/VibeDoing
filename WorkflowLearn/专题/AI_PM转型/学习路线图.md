# AI PM 学习路线图

> 最近确认时间：2026-04-16
> 状态：正式学习方案

## 目的

这份文档是当前仓库里关于 `AI 应用 PM + Agent / Workflow PM` 的正式学习路线图。
它把之前两份分析文档收敛成一条可执行主线，并明确现有项目如何映射进来。
全景式目标产出另见 `全景预期产出清单.md`。

## 使用原则

- 先学共同主干，再做系统化加深，最后补产品化作品。
- 课程内容优先拆成稳定概念、时效内容、验证记录、全景产出。
- 厂商快照、价格、窗口、默认值不进主干，只进 `volatile` 或刷新队列。

## 需要补强的横向能力

这些内容不单独拆成基础主线，但应显式嵌入后续模块。

### 1. Generative UI / AI-native 交互

- 归入 `M10`。
- 不只学聊天框，还要覆盖 `copilot`、`inline edit`、`ghost text`、侧边栏协作和 tool-driven UI。
- 失败体验不能只停留在 `M5` 的抽象层，还要落实成：失败提示、引用透出、人工接管入口、可恢复交互状态。

### 2. Token 经济学 / 定价 / ROI

- 归入 `M11-M13`，不前置为独立基础课。
- 学习重点不是背某家模型价格，而是会做：单次任务成本拆解、缓存/上下文/工具调用的成本影响、B 端 ROI 论证、seat vs usage vs outcome 定价取舍。
- 这是一条产品化与商业化能力，而不是先修技术主干。

### 3. 数据飞轮与 eval dataset 运营

- 强化到 `M4` 和 `M9`。
- 要学会把显式反馈、隐式修改行为、失败案例和人工复核结果沉淀成 eval dataset。
- `continuous eval` 不是一次性上线检查，而是上线后的长期数据循环。

### 4. Workflow 编排范式与 DSL / Schema

- 强化到 `M6` 和 `M8`。
- 要区分 `DAG / 明确编排` 与 `agentic planning / 动态决策` 的适用边界。
- 要能设计让模型和系统都稳定执行的 `JSON schema`、tool contract 和状态传递结构。

### 5. 人机协作与组织落地

- 强化到 `M5`、`M11-M13`。
- 要学会设计：人工复核位、升级路径、责任边界、审批点、失败回退和 SOP 改造。
- 对 `Workflow / Agent PM` 来说，产品成功往往不只取决于模型能力，还取决于是否真的嵌进团队日常流程。
