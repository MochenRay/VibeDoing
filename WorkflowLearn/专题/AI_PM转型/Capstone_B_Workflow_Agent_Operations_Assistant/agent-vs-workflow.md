# Agent vs Workflow Memo

> 最近确认时间：2026-04-16
> 场景：内部运营工单协同助手
> 结论：建议采用 `workflow-first + agentic substeps`

## 问题定义

内部运营团队每天会收到大量跨系统工单，例如：

- 查询用户状态
- 检查政策或运营规则
- 拉取 CRM / 工单历史
- 生成建议处理方案
- 在高风险场景里升级给人工审批

## 为什么不是纯 workflow

- 工单描述不标准
- 所需信息分布在多个系统
- 有些步骤需要在自然语言里做判断和摘要
- 同一问题的处理路径不完全固定

## 为什么也不是全自由 agent

- 大量环节有明确 SOP
- 一些工具调用具备写入副作用
- 审批边界和权限边界不能交给模型自由决定
- 可观测、恢复和值班要求很高

## 判断

最合理的方案是：

- 主路径用显式 workflow 固定住
- 把局部需要动态判断的地方交给 agentic substeps
- 高风险节点必须回到规则流和人工审批

## 什么时候升级成更强 agent

- 工具数增多
- 路由复杂度提升
- 任务需要跨轮规划
- 明确 workflow 无法覆盖的例外场景持续增加

## 决策

- 主方案：`workflow-first + agentic substeps`
- 必需条件：
  - 工具边界清楚
  - 状态可恢复
  - handoff 可观察
  - 高风险节点有人审
