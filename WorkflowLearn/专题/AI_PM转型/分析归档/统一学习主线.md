# 统一学习主线：AI 应用 PM + Agent / Workflow PM

更新时间：2026-04-16

## 1. 先说结论

你的理解 **大体正确，但不能简化成“只是顺序不同”**。

更准确的说法是：

- **底层主干内容高度重叠**
- **岗位分化主要发生在深度、侧重点和最终作品**

所以如果目标是“取二者并集，一起学”，最合理的方式不是维护两套课程，而是：

**一条统一主线 + 两块加深模块**

## 2. 二者到底哪里相同，哪里不同

### 2.1 相同的部分

不管你最终投的是 `AI 应用 PM` 还是 `Agent / Workflow PM`，都必须掌握：

1. 大模型的基本运行边界
2. Prompt / Schema / Tool Use
3. Retrieval / RAG / Grounding
4. Evals / Quality / Rollout
5. HITL / Fallback / 风险控制

这是共同底盘。

### 2.2 不同的部分

#### AI 应用 PM 更强调

- 用户价值
- 场景设计
- 体验设计
- 功能上线判断
- Fallback / 置信度 / 人机协同
- 商业价值和 adoption

#### Agent / Workflow PM 更强调

- Orchestration
- State / memory / context
- Tool contracts / MCP / connectors
- Workflow reliability
- Observability / tracing / telemetry
- Runtime / platform thinking

所以差别不是“学的完全不同”，而是：

- **AI 应用 PM 更像做 AI feature 和产品化**
- **Agent / Workflow PM 更像做 AI system 和运行层**

## 3. 统一学习主线怎么设计

建议用 3 层结构：

1. **共同主干**
2. **产品化加深**
3. **Agent / Workflow 加深**

### 3.1 共同主干

这部分必须全部学。

#### 模块 A：LLM 基础边界

目标：

- 理解模型为什么会错
- 理解 token / context / sampling / hallucination 对产品设计的影响

核心内容：

- Token
- Context Window
- Temperature / sampling
- Hallucination
- “模型是预测系统，不是数据库”

来源映射：

- 现有 `No.1` 的可保留部分

#### 模块 B：Prompt、Schema、Tool Use

目标：

- 从“写 prompt”升级到“约束模型行为”

核心内容：

- Prompt 基本结构
- System instructions
- Few-shot
- Structured Outputs
- Function calling / tool use
- Prompt 迭代方法

来源映射：

- 现有 `No.4`

#### 模块 C：Retrieval 与知识接入

目标：

- 理解知识如何被模型接入并稳定使用

核心内容：

- RAG 流程
- Embedding
- ANN
- Chunking
- Retrieval failure vs generation failure
- Grounding / source attribution

来源映射：

- 现有 `No.2`

#### 模块 D：Evals 与上线判断

目标：

- 把 AI 能力变成可评估、可上线、可回滚的系统

核心内容：

- representative samples
- ground truth
- metric-based eval
- human eval
- LLM-as-judge
- regression / drift
- launch criteria / rollback criteria

来源映射：

- 重写后的 `No.3`

#### 模块 E：产品化安全层

目标：

- 理解 AI 产品不是“能答出来”就结束

核心内容：

- Fallback
- HITL
- 置信度表达
- expectation management
- escalation
- failure handling

来源映射：

- 现有 `No.5` 的可保留部分

### 3.2 产品化加深

这是偏 `AI 应用 PM` 的深挖层，但建议一起学。

#### 模块 F：AI 功能设计与体验

核心内容：

- AI feature discovery
- feature scoping
- 用户心智与 trust design
- AI output 的展示方式
- 结果解释与引用
- adoption / retention / conversion

#### 模块 G：垂直场景方案设计

核心内容：

- 企业知识助手
- 对话式分析
- 智能摘要 / extraction
- workflow copilot
- 行业垂直 AI assistant

#### 模块 H：AI 应用 PM 作品输出

最终产出：

- 1 份 AI feature PRD
- 1 份 eval plan
- 1 份 launch / rollback 决策文档
- 2 个垂直场景案例拆解

### 3.3 Agent / Workflow 加深

这是偏 `Agent / Workflow PM` 的深挖层，但也建议一起学。

#### 模块 I：Agent Fundamentals

核心内容：

- agent vs workflow vs single-shot
- ReAct
- planning loops
- role decomposition
- single-agent vs multi-agent

来源映射：

- 现有 `No.6` 的基础部分

#### 模块 J：State、Memory、Long-running Tasks

核心内容：

- short-term context
- memory
- progress artifacts
- handoff
- session continuity
- long-horizon tasks

#### 模块 K：Tool Contracts、MCP、Connectors

核心内容：

- tools as contracts
- parameter correctness
- function calling discipline
- MCP
- connectors
- external system boundaries

#### 模块 L：Runtime、Observability、Reliability

核心内容：

- workflow reliability
- tracing
- logging
- telemetry
- failure recovery
- quality gates
- production rollout

#### 模块 M：Agent / Workflow PM 作品输出

最终产出：

- 1 份 agent workflow design doc
- 1 份 orchestration spec
- 1 份 eval matrix
- 1 份 runtime / telemetry / failure-handling 设计稿

## 4. 统一后的课程排序

如果你要把两条路线的并集一起学，建议顺序是：

1. **LLM 基础边界**
2. **Prompt、Schema、Tool Use**
3. **Retrieval 与知识接入**
4. **Evals 与上线判断**
5. **产品化安全层**
6. **AI 功能设计与体验**
7. **Agent Fundamentals**
8. **State / Memory / Long-running Tasks**
9. **Tool Contracts / MCP / Connectors**
10. **Runtime / Observability / Reliability**
11. **垂直场景方案**
12. **双路线作品集输出**

这个顺序的逻辑是：

- 先学“模型边界”
- 再学“怎么约束模型”
- 再学“怎么接知识”
- 再学“怎么评估是否能上线”
- 然后补“产品化设计”
- 最后进入 agent / workflow 的系统层

## 7. 我对你的建议

如果你的目标是“同时覆盖 AI 应用 PM 和 Agent / Workflow PM”，最优策略不是两边平均发力，而是：

1. **先用共同主干建立硬地基**
2. **优先把 Agent / Workflow 方向做深**
3. **再把 AI 应用 PM 的产品化能力补齐**

原因：

- 共同主干本来就是大头
- Agent / Workflow PM 对系统能力要求更高
- 一旦你能讲清楚 tools、state、evals、workflow reliability，再往 AI 应用 PM 迁移很容易
- 反过来只会做 AI feature，往 agent platform 走会更吃力

## 8. 一句话结论

你的理解可以修正为：

**两类岗位不是学完全不同的东西，而是学同一套 AI 产品基础设施；区别只在于一个更偏产品化，一个更偏运行时与编排层。**

所以最合理的路线确实是：

**取二者并集，一起学，但按“共同主干 → 系统化加深 → 产品化补全”的顺序推进。**
