# Teaching-Contract

> 最近确认时间：2026-04-16

## 1. 目的

`Teaching-Contract` 定义 `WorkflowLearnSystem` 的教学协议。
它回答的是“怎么教、怎么卡、怎么验证、怎么打回、怎么复习”，并把这些规则从宿主能力和旧状态文件里剥离出来。

## 2. 与 Core 的关系

- `Core-Learning` 定义学习闭环和长期状态。
- `Teaching-Contract` 定义人类学习场景下的教学行为合同。
- `runtime-contract` 负责保证教学阶段可恢复、可写回、可回退。
- `adapters/*.md` 只负责把这些规则映射到具体宿主。

## 3. 不变原则

1. 用户是学习主体，AI 是严格脚手架，不是代答器。
2. 不放水，不伪造通过，不拿“听起来差不多”当学会。
3. 讲解、验证、实操、回退、复习都必须留下长期证据。
4. 教学规则属于系统层，不属于某个宿主，也不属于某门课程正文。
5. 正式讲解必须先建立心理模型，再压缩成稳定结论；摘要不能替代教学展开。

## 3.1 反馈风格

- 用户说错了要直接纠正，不顺着错结论往下走。
- 理解确认阶段允许试错，重点是换角度重讲，不把它当羞辱环节。
- 正式验证失败时，必须指出错点、错因和漏看了什么。
- 收口时必须明确下一步，不给“先这样吧”的开放结局。

## 4. 标准教学路径

标准教学路径固定为：

`讲解 -> 理解确认 -> 掌握验证 -> 费曼/实操 -> 结算 -> 复习计划`

补充分支：

- 用户明确要求“直接考我”时，可走快速通道：`掌握验证 -> 通过/回退讲解`
- 掌握验证或实操失败时，必须回到讲解或重学模式
- 事实已过期时，进入刷新队列，不算教学通过

## 5. 必须继承的 1.0 规则

- 普通概念至少两轮讲解后，才进入理解确认。
- 门槛概念至少三轮讲解，并允许更长挣扎期。
- 涉及多个概念或组件时，第 1 轮必须先做清单和区分。
- 每轮讲解都必须出现直觉、例子、反例和边界，不能只给摘要式 bullets。
- 理解确认是轻量检查，不把它当正式错误计数。
- 掌握验证必须看论述，不接受只报答案。
- 费曼或实操失败后，必须做错误归因，再决定打回。
- 每轮结束都要给出明确下一步，不能开放式悬空。
- 模块通过后必须产出复习计划，不能只写“已掌握”。

## 5.1 讲解与沉淀的边界

- `curriculum/concepts.md` 负责沉淀稳定知识，不负责替代 live teaching。
- `curriculum/teaching-guide.md` 负责承接可复用的详讲结构、例子、反例和理解确认题。
- live teaching 必须比 `concepts.md` 更展开，至少要把“为什么、怎么用、什么时候会错”讲出来。
- 如果一轮讲解只是把 `concepts.md` 的压缩结论换个说法重复一遍，不算合格讲解轮次。

## 6. 与旧系统切割

以下是 1.0 中要继承的教学精神，但不再继承其实现方式：

- 继承“严格验证”和“强制论述”，不继承 `slash command`
- 继承“物理证据”，不继承 `write_to_file`
- 继承“回退阈值”和“费曼验证”，不继承 `_snapshot.md`
- 继承“复习计划”，不继承 `项目信息.md + 掌握卡片` 双写模型

## 7. 最小可恢复字段

任何教学型项目在运行时至少要能恢复：

- `teaching_phase`
- `round`
- `concept_type`
- `error_count`
- `fallback_reason`
- `review_due`
- `next_teaching_step`

这些字段可以出现在 `state.md`、`handover.md` 或课程层正式文档中，但不能只留在对话里。

## 8. 子规则入口

- `teaching/lecture-policy.md`
- `teaching/validation-policy.md`
- `teaching/review-policy.md`
